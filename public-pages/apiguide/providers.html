Provider keys + integration patterns

This page summarizes how API keys and provider settings are expected to work in the InnerAnimal Media stack (Agent Sam dashboard + Worker APIs). It’s written to be safe-by-default: store secrets in the vault, never paste them into code, and rotate fast when in doubt.

Where keys live

In production, user-scoped LLM keys are stored encrypted in the vault and used via the Worker session. In the dashboard, manage keys at Settings → Security.

Vault: /api/vault/store, /api/vault/llm-keys Scope: session user Encryption: AES-GCM 
# Dashboard (recommended)
Open: /dashboard/settings → Security

# API (advanced)
GET    /api/vault/llm-keys
POST   /api/vault/store                 { key_name, value }
DELETE /api/vault/llm-keys/:id

Quick safety checklist

  • Never commit keys (no `.env` / plaintext in repo).
  • Prefer vault storage (encrypted) over env vars for user keys.
  • Rotate keys immediately if exposed; revoke old keys.
  • Use least privilege scopes where providers support it.
  • If a request fails: check key validity, rate limits, and provider outages first.

Providers

Expected key formats and typical usage notes.

Anthropic

Vault slot: ANTHROPIC_API_KEY Prefix: sk-ant-…
  • Use for Claude chat + reasoning tasks.
  • Store in vault via the Security tab; the UI masks the value.
  • Rotate if leaked; treat keys as prod credentials.

OpenAI

Vault slot: OPENAI_API_KEY Prefix: sk-…
  • Use for embeddings, tools, and general-purpose chat when configured.
  • Mind rate limits; use batching where possible.

Gemini

Vault slot: GEMINI_API_KEY Prefix: AIza…
  • Use for Google GenAI models and multimodal workflows.
  • Keep project restrictions tight; rotate on leaks.
